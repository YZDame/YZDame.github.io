2026年3月AI工具使用小结与个人项目复盘

发表于 2026-04-05 更新于 2026-04-07 本文字数： 5.2k 阅读时长 ≈ 10 分钟

事情是这样的。

我本来以为，2026 年 3 月会是一个安安静静补项目的月份。

结果并没有。

这个月给我的真实体感，不是在用工具，而是在被工具推着跑。

前脚一个套餐降级，后脚一个模型升级。上午还在想，哎这个配置终于顺手了，下午一看，又变了。你敢信？？？

所以我后来索性不去追那种很宏大的判断了，不聊什么行业趋势，不聊什么颠覆不颠覆。回到我自己这块，这一个月我更关心一件事，哪些东西我真的在高频用，哪些项目真的往前拱了一步，哪些看起来很热闹，其实离我还远。

先说模型这块。

三月最让我有体感的，不是哪个模型突然封神，而是有些东西真的会悄无声息地给你降配。

Copilot 教育版就是一个很典型的例子。

原本还能用 Opus 4.6，结果三月初突然就没了，只剩下 Haiku 4.5 和 Codex 5.3。这个变化看着像一行配置改动，落到使用上其实挺伤的。尤其是你已经适应了某个模型的思考密度和写作手感，再被切回去，那个落差不是参数表能解释的，就是一句话，不顺手了。

我当时就愣住了。

因为这种体验很像什么呢，很像你常去的一家小店，老板没跟你打招呼，默默把原来的肉换了一种。你第一口就能吃出来，但又说不出具体是哪儿不对。不是完全不能吃，而是你知道，那个熟悉的感觉回不来了。

回到 ChatGPT Plus 这块，倒是另一种极端。

gpt-5.4 fast 真的很快，快到有时候让我产生一种错觉，感觉自己脑子也跟着变快了。很多本来会卡一下的地方，丢进去马上就给反馈，写代码、改文档、顺结构，整体都很丝滑。

但速度这个事吧，从来都不是白拿的。

它快，token 也跑得快。你如果最近一直在高强度对话，那个消耗感是很明显的。很多朋友可能不知道，模型用久了以后，你对一个工具的评价已经不只是好不好用，而是值不值得这样用。快当然爽，可如果每一次爽都对应更快的消耗，你脑子里就会自动开始算账。

GLM 这边，我年初买过一个最便宜的 Lite 包季，四十多块钱。最近套餐快到期了，GLM 4.7 也切到了 GLM 5.1。坦率的讲，5.1 比之前是好一些的，这个进步我能感觉到。

但问题也很现实。

我没那么常用它了。

这不是它不行，而是我的工作流已经被别的工具占住了。工具世界有时候挺残酷的，你只要没进入一个人的主流程，就很容易被放到边上。能力变强是一回事，被不被想起来又是另一回事。

MiniMax M2.7 则是另一个让我觉得有点意思的存在。

它在一个多月里从 M2.1 升到 M2.7，能力是能感受到在长的，而且包年 200 元这个价格，说真的，性价比已经挺猛了。我现在把它当 OpenClaw 的主力模型在用，很多事情它都能扛。

但 Agents 执行这块，它还差点火候。

这种差距很微妙，不是你看一眼结果就会说它不行，而是你真的把任务链跑起来，才会发现它在一些需要持续规划、持续校准的地方，还是会发飘。愚钝如我，这种坑踩过几次之后就会明白，模型排行榜和你真实拿来干活，完全是两码事。

所以如果你问我，三月我对模型最大的感受是什么。

不是谁最强。

而是稳定、顺手、进入主流程，这三件事比一切纸面参数都重要。

顺着上面的再聊聊项目。

这个月我最开心的，不是看了多少新发布，而是一些拖了很久的想法，终于开始长出具体形状了。

比如 WeMath2MD。

这个项目其实三年前我就有念头了。因为我一直对微信公众号里的数学内容有个执念，很多文章明明信息密度很高，到头来却还是图片拼接。你想引用，没法改。你想检索，检不了。你想继续加工成讲义、笔记、题库，更麻烦。

以前这事基本是体力活。

现在不一样了。

我在终端里直接敲 WeMath2MD，后面跟一个公众号链接，它就会调用 MinerU API 去下载图片、做解析，然后把 OCR 的结果汇总成 Markdown，图片按 Markdown 外链的方式塞回文档里。你如果一直在处理这种材料，就会明白这个变化有多爽。原来那种只能看、不能用的内容，突然就进入可编辑、可检索、可再创作的状态了。

这玩意让我一直很兴奋的一点是，它不是一个孤零零的转换器。

它其实是在把一类本来被锁死的信息，往开放格式里搬。

我有时候觉得，AI 真正让人上头的地方，不是它替你写了一段什么漂亮的话，而是它开始帮你把过去那些根本懒得碰、也没法碰的脏活累活给拆了。门一开，后面的工作流才有机会成立。

Gikz 也是类似的感觉。

它的起点是我刷到 B 站一个视频，GeoTikTirm。看完以后我脑子里冒出来的，不是这个效果真炫，而是另一个问题，Geogebra 导出来的 GGB 代码，能不能被清洗得更像人写的东西。

然后我就顺手下场了。

Gikz 现在做的事，说到底就是清洗 Geogebra 导出的那些复杂、凌乱、读起来脑壳疼的代码，把它们整理成更简洁、更好看的形式。我给它做了网页版，也发了 npm 包，全局装完之后随手就能用。

很多项目在介绍的时候都爱说自己解决了什么痛点。

我对这种话现在比较警惕。

但 Gikz 确实帮我打开了一个新的口子，因为它不是终点，它后来直接把我带进了 tsqx 这条线。

tsqx 是我三月很重要的一条主线。

如果你平时不碰几何绘图，可能会觉得这名字有点怪。简单讲，它可以理解成一种比 Asymptote 更轻一点、写几何关系更直观一点的表达方式。尤其是平面几何，很多关系一旦换一种写法，脑子里那根筋是会突然通的。

我之所以会继续往这条路上挖，很大程度上也是因为又去翻了 Evan Chen 的主页。他的很多讲义里，大量在用 Asymptote 或 tsqx 画图。你看久了会有一种感觉，好的工具不仅在帮你出图，它还在反过来塑造你理解图形关系的方式。

这个感觉很妙。

所以我没有只停在「我自己会用」这一步。

目前 tsqx 我已经陆续做了三样东西，中英文说明书，VS Code 扩展，还有能在 OpenClaw、cc、Codex 里调用的 skill。前两个更像基础设施，后一个才是我最近越来越在意的方向。

因为工具一旦进入 Agent 体系，味道就变了。

它不再只是一个人点开来用一下的东西，而是有机会嵌进一整条任务链里。你给它一个目标，它和别的组件开始互相调用，开始协作，最终生成一个你本来要手搓很久的结果。

这一下给我更干懵了。

我慢慢意识到，我这半年其实一直在做同一件事，把原本分散在不同软件、不同窗口、不同脑内操作里的步骤，尽可能折叠成一条可调用的链。

GeoGebra-MCP-Server 就是这个思路下长出来的。

它的起点，是我看了 MathOver 的视频，然后去折腾一个开源的 GeoGebra MCP 项目 Gebrai。我对它做了样式调整和重命名，也发了 npm 包，现在在 cc、Codex 这类工具里都能调。

这个项目最让我兴奋的，不是改名成功，也不是打包上线，而是它验证了一件事。

Coding Agents 真的可以通过 CLI 去调工具链，然后生成 GGB 文件。

这句话看着有点技术宅。

但你把它翻成人话，其实很简单，很多原本以为必须靠手工点点点、或者必须做成重后端服务的事情，现在未必了。你可能只需要把工具打磨到能被 Agent 调用，很多复杂度就被转移掉了。

当然，商业化应用也不是不能做。B 站那位 up 主已经在做了，只是那条路要处理服务器、账户、缓存数据，还有一些订阅付费逻辑。那是另一种工程，没什么高下之分，只是重量级完全不同。

我自己现在更想站在另一边。

先把轻的、快的、可组合的这条路走明白。

回到题库这块，我三月还被 B 站推了一个视频，奇思妙想LaTeX 题库。

然后我又想起 Evan Chen 的 Von 题库。

这东西是真的强，组卷、引用题目、组织材料，整个思路都非常成熟。于是我脑子里立刻又冒出来一个比较骚的事，能不能把这套题库镜像成 Markdown 形式的 Obsidian vault，然后慢慢接到 Agent 自动化流程里。

你看，又回到那个老问题了。

为什么我最近总想把东西往 Markdown 里搬。

因为 Markdown 在我这里，已经不只是一个写文章的格式了。它越来越像一种中间层，一种让知识、工具、脚本、Agent 能彼此说话的公共语言。你把资料困在 PDF、图片、专有数据库里，它们就像一堆彼此看不见的孤岛。可一旦进了 Markdown，很多事情突然就活了。

说到这个，OCR 这块我这个月也挺上头。

尤其是 Mistral OCR。

这玩意拿来处理 PDF 扫描版书籍或者文章，真的很顺。速度快，免费，准确率也不错。大概半分钟左右，一份扫描材料就能进 Markdown。

但真正有意思的，不是 OCR 本身。

而是 OCR 之后你终于能开始干活了。

我跟你说，很多时候大家把 OCR 理解成提取文字，这个理解太浅了。OCR 更像是把一份原本只能被人眼浏览的东西，变成一份可以被整个工具链继续加工的材料。后面你让 Codex 或 Claude Code 去修结构、修公式、修语法，再丢进 Obsidian 里预览、做笔记、写总结，整个链条就接上了。

我这个月就拿一本数论书的 PDF 扫描版试过。

先用 Mistral OCR 转成 Markdown，再用 Codex 修一部分符号问题，再在 Obsidian 里整理欧拉函数的证明，写成一份小讲义。这个过程本身就很能说明问题，你以前面对一本扫描书，更多是读。现在你面对它，已经开始改、开始拆、开始重组、开始二次生产了。

这不是小变化。

这会直接改变一个人跟知识材料相处的姿态。

再往工作里落一点。

我三月还在折腾学生数据库这件事。用 OpenClaw、Claude Code、Codex，再配合飞书或者 Notion，去搭一套学生信息和成绩的结构化系统。

坦率的讲，这玩意推广给其他老师用，现在还有难度。

我非常理解这种难度。

你不是天天折腾命令行的人，你也不是天天在折腾自动化的人。你就是一个正常忙碌的老师，手上课多、杂事多、时间碎。这个时候让你突然改习惯，去信任一套新流程，本来就不容易。

但从我自己的感受看，底层逻辑已经跑通了。

一旦成绩表、学生信息、过程记录可以被比较稳定地结构化，后面很多事都会自然长出来。检索、汇总、追踪、统计，都会轻很多。不是说一下子全自动了，而是你终于不用每次都从零开始搬砖。

Office 这块也一样。

我这个月越来越常让 Claude Code、Codex、MiniMax 去碰 docx、xlsx、pptx。以前这类文件是很重的黑盒，很多操作你默认只能手动。现在情况不一样了，至少在一部分场景里，它们已经开始变成可以被脚本和 Agent 驱动的对象。

还有一个我自己都觉得有点离谱的使用方式。

直接让 ChatGPT 生成 LaTeX 小讲义。

有时候赶时间，我就直接在对话框里让它吐源代码，然后丢到 Overleaf 编译。以前这件事还带点实验性质，现在我是真的会在需要的时候这么干。甚至现在 ChatGPT 还能后台跑，直接给源代码和 PDF 下载链接。

这会带来一个特别具体的变化。

你不一定非要坐在电脑前，手机上也能瞬间把一个小讲义生出来。

很多朋友可能会觉得，这不就是方便一点吗。

我自己的感受不是方便一点，而是表达和生产之间那层摩擦，正在被一点点磨薄。以前你脑子里有个教学念头，中间要跨很多工具、很多格式、很多等待。现在这些阻力虽然没消失，但确实在变小。

再说一个我觉得挺私人，但又很能代表这个月状态的项目，Paralex。

这是我一次很典型的 Vibe Coding 实践。

我想做一个原生 macOS 应用，去实现双语对照翻译，主要用来看英文论文和 PDF。整个思路很朴素，左边导入 PDF，右边做 OCR 和翻译，再提供不同语言的输出，还有原文、译文、双语版本的下载。

它离真正投入生产还远。

说实话我们还差得远。

但我还是很喜欢它。

因为它特别像这个月所有事情的一个缩影，你先是被一个很具体的需求刺到，然后鬼使神差地开始动手。动手以后发现问题一堆，细节一堆，坑也一堆。可正因为你真的下场了，那个想法才第一次从脑子里跑到了屏幕上。

我一直觉得，2026 年做这些项目，最珍贵的已经不是单点能力，而是把想法迅速压到原型、再把原型接进工作流的能力。

工具会变，套餐会砍，模型名字会一版一版更新。

但一个人如果慢慢学会了怎么把零散的需求，折叠成一条自己能复用的链，那很多事情就会开始不一样。

这也是为什么，我三月虽然看起来做的东西很散，OCR、题库、几何绘图、数据库、Office、原生应用，哪边都沾一点，但在我自己心里，它们其实是一回事。

都是在试，怎么把过去那些卡住我的摩擦，磨掉一点。

磨掉一点，再磨掉一点。

然后让更多原本只能想想的东西，真的跑起来。

大时代啊，朋友们。

如果说这个月有什么特别强的结论，我还真不敢下。

我自己也还在摸索。

但有一个感觉越来越清楚了，未来真正有意思的人，未必是知道最多工具的人，而是能把几个看起来很普通的工具，捏成一套自己工作流的人。

这个人可能不是程序员，不是全职开发者，也不是什么 AI 博主。他可能只是一个老师，一个内容创作者，一个需要长期处理知识材料的人。

可一旦他把这条链捏出来，很多事情就会变。

反正我觉得，这事挺值得继续搞的。

以上，既然看到这里了，如果觉得不错，随手点个赞、在看、转发三连吧，如果想第一时间收到推送，也可以给我个星标⭐～
